# Original Brain Dump for agent-ledger

## This is the initial stream-of-consciousness document I wrote when coming up with the idea for agent-ledger. The scope and architecture has changed since this was written, but it is preserved here for agents to be aware of my original intentions and goals.

1. Look for existing tools that can do this
2. Investigate what's possible for having a script detect claude sessions running in worktrees, whether we can see their names, whether we can inject prompts into them, whether we can see if they need input, whether we can summarize them
3. Examine how we can have worktrees aware of the ids of their Claude sessions for easy tracking and resuming from the repl, inline or in cmux
4. Examine how we can enhance the existing worklog to be worktee-aware and claude-aware
5. Develop a skill so Claude automatically periodically logs what it's doing on its branch/worktree, with Jira and GitHub context. Concise current summary, and full timestamped log of activities. Rolling, to preserve context. Maybe daily summaries and separate detailed summaries. Should track what we're doing, problems we identify, plans we make (and references to plan file backups), observations we make about external references like PRs, issue comments, associated slack threads (attach them to worktrees). Centralize a record of active and archived worktrees or make it easy to look them up.
6. Make an agent-facing CLI for accessing data for context on these and potentially for updating them. Maybe just adapt the worktree script to be able to handle this.
	1. Perhaps make the worktree script a little more terminal driven instead of repl driven. Tab completion etc.
7. Enhance the worktree script to archive rather than delete old worktrees. Or when deleting, archive and retain the logs
8. Aggregate the worklog-specific logs to a combined timestamped log, either automatically or on demand. Maybe duplicate or maybe just be able to query. Are we having the agents read markdown or request this context via CLI? We could use a database, but possibly overkill. Maybe split up files more instead and index them.
9. Have an orchestration agent that is aware of these logs being populated automatically by other agents. The agents should be reporting what they're doing and seeing. The orchestrator should ideally be able to inject prompts into the others, or send them tool call approvals, or ask them to report their status, etc. maybe trigger a /btw and get its response.
10. Explore agent archetypes and how they can fit together with this tooling.
	1. The main agents I interact with in each worktree are engineers on a team, either in developer role or reviewer role
	2. There could be a team lead agent responsible for checking in with them on status and logging high level status across them (will have to see if that's better than having each engineer agent log its own activity or we want some combination of them, like engineers sending those reports to the lead agent and having the lead write those memories).
11. We could have a GitHub and Jira watcher agent responsible for being aware of the PRs and issues that are actively being worked on in worktrees, and checking their status to look for things like new review comments and CI results on my PRs or new commits and comments on PRs I'm reviewing. Perhaps the developer and reviewer agents tell the watcher agent when they push new commits so the watcher agent is aware of what it should be watching. Either way, the watcher agent should just periodically check worktrees to see what commits they have and what PRs and issues they are involved with, and then check those external resources for updates. When it sees new updates on them it could inform the worktree-specific engineer and reviewer agents so they can take action. This watcher agent would run autonomously, I'd need to figure out how to make that work and how to track what's going on. The team lead agent could maybe be my interface to the watcher agent. When the watcher agent is checking for PRs and issues associated with a worktree or checking for new activity on those resources, it should do so with sub agent so it doesn't use up its context too much. But as a long running agent, it should be aware of its context window and make sure it is persisting a log/status (or updating the shared logs/status the worktree agents are using) so that we can restart that agent and it'll pick up where it left off. But I'll need to rotate and archive those logs to not overload context, and be able to look up archived history on demand without it automatically cluttering things.
12. We will need clean abstractions for the agent-facing commands of the worktree script for operations like asking for worktree metadata (branch, associated issue and PRs, available logs/context, active and archived Claude session IDs). We'll also need clean abstractions for cross-agent communication, which will require investigating what is possible for communicating with a running Claude code CLI session. Some of this will need to be scripts, and some will need to be skills so the agents know about the scripts and about each other.
13. I really need to investigate what tools already exist for this so I'm not reinventing the wheel, but I do still want to be able to run each worktree's Claude code sessions in their separate cmux terminal workspaces, and just be able to somehow connect them together and have them assembled into a team that can communicate and share information.
14. It would be nice to be able to connect remotely to the orchestrator/lead agent so I can ask it about the team's status from my phone.
15. It also might be cool to have a local web interface that reports what's going on with the agents and gives me quick access to them, their activity logs, their associated PRs (with status) and issues (with status), etc.
16. When an agent wants new information about an issue or PR it should be able to do so in a way that reuses and updates the shared context/cache/log. When it gains new information about status of these external resources it should be able to notify the agents that might benefit from having that up to date info.
17. The goal here is to be able to work on multiple things at a time and be able to always see the high level status of all that work, have the whole team of agents able to respond to what's going on with issues and PRs in an autonomous way, be able to ask about what they've been doing and what they accomplished each day, the highlights of problems they encountered and solved, the most important things I should know or care about from their work, etc. I want to automate the parts of the workflow that are just process gaps like tracking status and cross communication, but I still want full control of the work they are doing, I want to be able to review changes before committing and review conclusions before decisions are made, and retain control over tool call approval.
18. I want this to integrate well with my existing scripts like worktree and worklog, adapting them so they work better in this workflow but having them also be useful for my own manual use.
19. I want help thinking about anything I might be missing here. I don't want to just do exactly what I'm asking for if it's not the best way to solve this problem. But generally speaking I'm looking to level up how I use these tools and unlock some additional efficiency and organization of my day to day work.
20. One example of a prompt I want to be able to give my team lead / orchestrator agent is "what's the status of each worktree and what input is needed from me right now to progress each one towards its goal". I want to know where I need to jump in. I want to be able to stop working for a day, forget everything that's happening, come back tomorrow and ask my agent team the equivalent of "all right how are we doing, what should we do next". But I want all of that with me, the human, very much still in the loop where it matters. And I want to decide where that matters, and keep the risk low of AI making bad decisions based on hallucinations or misunderstandings.
21. I also want an even higher level view of what I should be doing to most effectively help my real human team. Maybe the orchestrator agent or maybe a higher level agent (one or more) should be able to look at what's going on with our in progress epics and what's ready to work on in them, help me identify priorities there to unblock others, help me identify issues with PRs that need review, help me identify conversations that need my input or that have new information for me (via slack threads associated with my worktrees either manually or via detection that a PR or issue was linked in a slack thread - perhaps we have a slack watcher agent. Also via email). Perhaps this higher view also includes ingestion of meeting summaries from Gemini that can be found in my email and correlates that with what I'm doing at a high level. Maybe I can ask the agent what's going on with my team, how I can help Yulia, how I can help Emily, etc.
22. A lot of this, especially that last higher level ambitious idea, requires a lot of context and comprehension that will be difficult to manage. I will need a good plan for making use of concise but thorough persisted context files, abstracting operations like checking status and cross agent communication into subagent tasks so they don't cause bloating of context, and proactive awareness of each session's context window so agents can hand off work to a fresh session where it makes sense to, etc.
23. I want to do all this in a way that is transparent and easy to track. I want to be able to see all my agents, what they're doing, how they are connected, what they're aware of and watching, how they're communicating. Maybe even with some visualization of my team in a live updating webapp or something.
24. Wherever possible, things that do not require LLM intelligence should be abstracted into deterministic scripts, as long as doing so doesn't introduce unnecessary complexity.
	1. It is okay to couple this workflow to tools I'm using like my work scripts and cmux.
25. I want to continue to iterate on and improve this setup, track how it's working, track the history of my usage of it and decisions about how to change how it works.
26. I want help anticipating problems with this setup and perhaps create a mechanism for diagnosing and correcting those problems. Things like context overload, duplicated state, out of date status information, redundant logging, inefficient communication, disorganization, sessions running too long, agents getting confused or caught in loops, etc. I should be able to regularly do a health check of the whole workflow.
27. I want to automatically maintain up to date documentation about how another person might set up a recreation of my setup.
28. I want to reiterate that it would be easy to reinvent the wheel and I want to do thorough research about what tools already exist for these use cases. I might want a combination of existing tools and my own bridges between those and my tools. If I need to compromise some of my requirements to avoid reinventing the wheel I might be ok with that. The first step in planning all this should be thorough research. I'm particularly curious what already exists for communication across multiple persistent Claude code sessions.
29. I might be overthinking all of this. Maybe I should start with something simple, or roll this out in a multi phase approach. Not all these ideas will work exactly how I want.
